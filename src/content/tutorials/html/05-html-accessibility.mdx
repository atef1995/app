## What is Web Accessibility? (And Why Should You Care?)

**Here's the thing:** When I say "people with disabilities," who do you picture in your mind?

Most people think of someone in a wheelchair, right? But **disability is much broader and more common than you might think.**

**Let's talk about real people:**
- **Sarah** is a brilliant developer who happens to be blind - she uses a screen reader to code
- **Mike** has ADHD and gets overwhelmed by busy, cluttered websites  
- **Maria** broke her wrist and can only use one hand to navigate
- **David** is colorblind and can't tell if that red button means "delete" or "save"

**Here's what might surprise you:** About 1 in 4 adults has some form of disability. That's **over 1 billion people worldwide!** 

**But here's the really cool part:** When you make websites accessible for people with permanent disabilities, you accidentally make them better for everyone else too!

## "But Why Should I Care?" - Let Me Give You Some GREAT Reasons! üí°

**Okay, I hear you thinking:** "This sounds nice, but I'm already overwhelmed learning to code. Why should accessibility be a priority?"

**Fair question!** Let me give you some answers that might change your mind:

### 1. It's the Law (And Lawsuits Are Expensive!) ‚öñÔ∏è

**Laws you should know about:**

- **AODA** in Canada - requires accessibility compliance
- **European Accessibility Act** - affects anyone serving EU users

**Translation:** Ignoring accessibility can literally cost you money. A lot of it.

### 2. It's Good Business (Ka-Ching!) üí∞

- **Better SEO rankings** - Google loves accessible websites

- **Positive brand reputation** - people notice when you care about inclusion
- **Lower bounce rates** - when people can actually use your site, they stay longer

**Real question for you:** Would you rather have a website that works for 75% of people, or 100% of people?

- **Be descriptive** but concise (aim for 125 characters or less)
- **Include important text** that appears in images
- **Describe function** for interactive images
- **Use empty alt** (`alt=""`) for purely decorative images
- **Don't start with** "Image of" or "Picture of"

## ARIA - HTML's Helpful Translator üó£Ô∏è

**First things first:** ARIA stands for "Accessible Rich Internet Applications" but let's forget that intimidating name and think of it differently.

**ARIA is like a translator** between your HTML and assistive technologies. Sometimes your HTML can't express everything you want it to, so ARIA steps in and says "Hey screen reader, let me explain what this element actually does!"

**Here's the key rule:** ARIA doesn't change how your website looks or behaves. It only changes how assistive technologies understand it.

**Think of ARIA like adding subtitles to a movie** - the movie doesn't change, but now more people can understand what's happening!

```css
/* ‚úÖ Good - 7.7:1 contrast */
.text { color: #333; background: #fff; }
```

**Pro tip:** Use WebAIM's contrast checker. Your eyes might think it's fine, but the math doesn't lie!

### 3. The Placeholder Text Trap üìù

**So tempting:** Just use placeholder text instead of labels - it looks cleaner! But then users can't tell what field they're in once they start typing...

```html
<!-- ‚ùå Bad -->
<input type="text" placeholder="Name">

<!-- ‚úÖ Good -->
<label for="name">Name</label>
<input type="text" id="name" name="name">
```

**Real talk:** Placeholder text disappears when you type. Labels stay visible. Which would you prefer?

### 4. The Fake Button Fiasco üé≠

**We've all been tempted:** "I'll just make this div look like a button!" But then keyboard users can't reach it...
```html
<!-- ‚ùå Bad -->
<div onclick="submit()">Submit</div>

<!-- ‚úÖ Good -->
<button type="submit">Submit</button>
```

### 5. Missing Focus Management
```html
<!-- ‚ùå Bad - No visible focus indicator -->
<button style="outline: none;">Click me</button>

<!-- ‚úÖ Good -->
<button class="has-focus-indicator">Click me</button>
```

### ‚úÖ HTML Structure

- [ ] Logical tab order

### ‚úÖ Images and Media

- [ ] Audio content has transcripts

### ‚úÖ Forms
- [ ] All inputs have associated labels
- [ ] Required fields are marked programmatically
- [ ] Error messages are clear and programmatically associated
- [ ] Form validation doesn't rely on color alone
- [ ] Submit buttons are clearly labeled

### ‚úÖ Keyboard Navigation
- [ ] All interactive elements are keyboard accessible
- [ ] Focus indicators are visible
- [ ] Skip navigation link provided
- [ ] Tab order is logical
- [ ] Keyboard traps are avoided

### ‚úÖ Color and Contrast
- [ ] Text meets WCAG contrast requirements
- [ ] Information isn't conveyed by color alone
- [ ] Focus indicators have sufficient contrast
- [ ] Interactive elements are distinguishable

### ‚úÖ ARIA and Semantics

- [ ] Landmarks identify page regions
- [ ] Dynamic content changes are announced
- [ ] Interactive widgets follow ARIA patterns
