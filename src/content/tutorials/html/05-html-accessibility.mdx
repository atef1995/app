# HTML Accessibility: Building Inclusive Web Experiences

Welcome to web accessibility! ‚ôø Accessibility ensures that websites work for everyone, including people with disabilities. It's not just about compliance‚Äîit's about creating inclusive experiences that benefit all users.

## What is Web Accessibility?

Web accessibility means designing and developing websites that can be used by people with various abilities and disabilities, including:

- **Visual impairments** (blindness, low vision, color blindness)
- **Hearing impairments** (deafness, hard of hearing)
- **Motor impairments** (limited fine motor control, paralysis)
- **Cognitive impairments** (dyslexia, ADHD, memory issues)

> **üí° Universal Benefit** Accessibility features often help everyone. Captions help in noisy environments, good contrast helps in bright sunlight, and clear navigation helps when you're distracted or in a hurry.

## Why Accessibility Matters

### Legal Requirements
Many countries have laws requiring digital accessibility:
- **Americans with Disabilities Act (ADA)** in the US
- **Accessibility for Ontarians with Disabilities Act (AODA)** in Canada
- **European Accessibility Act** in the EU

### Business Benefits
- **Larger audience** - 15% of the global population has some form of disability
- **Better SEO** - Many accessibility practices improve search rankings
- **Improved usability** for all users
- **Reduced legal risk** and compliance issues

## Semantic HTML: The Foundation

Semantic HTML provides meaning and structure that assistive technologies can understand:

<InteractiveCodeBlock title="Semantic vs Non-Semantic HTML" description="How semantic elements improve accessibility:">
{`<!-- ‚ùå Non-accessible HTML -->
<div class="header">
    <div class="nav">
        <div onclick="navigate()">Home</div>
        <div onclick="navigate()">About</div>
        <div onclick="navigate()">Contact</div>
    </div>
</div>
<div class="main">
    <div class="article-title">Breaking News</div>
    <div class="content">Article content here...</div>
</div>

<!-- ‚úÖ Accessible semantic HTML -->
<header>
    <nav aria-label="Main navigation">
        <ul>
            <li><a href="/">Home</a></li>
            <li><a href="/about">About</a></li>
            <li><a href="/contact">Contact</a></li>
        </ul>
    </nav>
</header>
<main>
    <article>
        <h1>Breaking News</h1>
        <p>Article content here...</p>
    </article>
</main>`}
</InteractiveCodeBlock>

### Why Semantic HTML Helps
- **Screen readers** can navigate by headings, landmarks, and lists
- **Keyboard users** can jump between interactive elements
- **Voice control** software can identify clickable elements
- **Search engines** better understand content structure

Headings create a document outline that screen readers use for navigation:

    </section>
    
    <section>
        <h2>Implementation Strategies</h2>

        <h3>ARIA Attributes</h3>
        <p>Enhancing semantics...</p>

<!-- ‚ùå Poor heading hierarchy -->
<article>
    <h1>Web Accessibility Guide</h1>
    <h3>Introduction</h3> <!-- Skips h2 -->
    <p>Content...</p>
    <h2>Implementation</h2>
    <h4>Semantic HTML</h4> <!-- Skips h3 -->

### Heading Best Practices
- **Start with h1** - Only one per page for the main title
- **Don't skip levels** - Go from h1 to h2 to h3, not h1 to h3
- **Use headings for structure**, not styling
- **Make headings descriptive** and meaningful

## Images and Alternative Text

Alternative text (alt text) describes images for users who can't see them:

<InteractiveCodeBlock title="Effective Alt Text" description="Writing meaningful alternative text for images:">
{`<!-- ‚úÖ Good alt text examples -->

<!-- Informative image -->
<img src="sales-chart.png" 
     alt="Sales increased 25% from Q1 to Q2, reaching $2.3 million">

<!-- Decorative image -->
<img src="decorative-border.png" alt="" role="presentation">

<!-- Functional image (button) -->
<a href="/search">
    <img src="search-icon.png" alt="Search">
</a>

<!-- Complex image with description -->
<figure>
    <img src="org-chart.png" 
         alt="Company organizational chart showing CEO at top, 
              three VPs below, and department heads under each VP">
    <figcaption>
        <a href="org-chart-details.html">View detailed organization chart</a>
    </figcaption>
</figure>

<!-- Image with text content -->
<img src="quote-image.png" 
     alt="Quote from Maya Angelou: 'If you don't like something, change it. 
          If you can't change it, change your attitude.'">

<!-- ‚ùå Poor alt text examples -->

<!-- Too generic -->
<img src="photo.jpg" alt="image">

<!-- Redundant -->
<img src="cat.jpg" alt="Image of a cat playing">
<!-- Should be: alt="Orange tabby cat playing with yarn ball" -->

<!-- Missing for informative content -->
<img src="important-chart.png" alt="">

<!-- Describing appearance instead of function -->
<a href="/download">
    <img src="button.png" alt="Green rectangular button">
    <!-- Should be: alt="Download report" -->
</a>`}
</InteractiveCodeBlock>

### Alt Text Guidelines
- **Be descriptive** but concise (aim for 125 characters or less)
- **Include important text** that appears in images
- **Describe function** for interactive images
- **Use empty alt** (`alt=""`) for purely decorative images
- **Don't start with** "Image of" or "Picture of"

## Accessible Forms

Forms are critical interaction points that must be accessible:

<InteractiveCodeBlock title="Accessible Form Design" description="Creating forms that work for everyone:">
{`<!-- ‚úÖ Accessible form -->
<form>
    <fieldset>
        <legend>Personal Information</legend>
        
        <!-- Proper label association -->
        <label for="first-name">First Name *</label>
        <input type="text" 
               id="first-name" 
               name="firstName" 
               required 
               aria-describedby="first-name-help"
               aria-invalid="false">
        <div id="first-name-help" class="help-text">
            Enter your legal first name
        </div>
        
        <!-- Email with validation -->
        <label for="email">Email Address *</label>
        <input type="email" 
               id="email" 
               name="email" 
               required 
               aria-describedby="email-error"
               aria-invalid="false">
        <div id="email-error" role="alert" class="error-message" style="display: none;">
            Please enter a valid email address
        </div>
        
        <!-- Radio button group -->
        <fieldset>
            <legend>Preferred Contact Method</legend>
            <input type="radio" id="contact-email" name="contact" value="email">
            <label for="contact-email">Email</label>
            
            <input type="radio" id="contact-phone" name="contact" value="phone">
            <label for="contact-phone">Phone</label>
        </fieldset>
        
        <!-- Complex input with instructions -->
        <label for="password">Password *</label>
        <input type="password" 
               id="password" 
               name="password" 
               required 
               minlength="8"
               aria-describedby="password-requirements"
               aria-invalid="false">
        <div id="password-requirements">
            Password must be at least 8 characters and include:
            <ul>
                <li>One uppercase letter</li>
                <li>One lowercase letter</li>
                <li>One number</li>
                <li>One special character</li>
            </ul>
        </div>
    </fieldset>
    
    <!-- Clear error messaging -->
    <div role="alert" id="form-errors" style="display: none;">
        <h2>Please correct the following errors:</h2>
        <ul id="error-list"></ul>
    </div>
    
    <button type="submit">Create Account</button>
</form>

<!-- ‚ùå Inaccessible form -->
<form>
    <!-- Missing labels -->
    <input type="text" placeholder="First Name">
    
    <!-- Poor error handling -->
    <input type="email" style="border: 1px solid red;">
    <span style="color: red;">Error</span>
    
    <!-- No grouping for related fields -->
    <input type="radio" name="contact" value="email"> Email
    <input type="radio" name="contact" value="phone"> Phone
</form>`}
</InteractiveCodeBlock>

### Form Accessibility Checklist
- **Associate labels** with inputs using `for` and `id`
- **Group related fields** with `<fieldset>` and `<legend>`
- **Provide clear instructions** using `aria-describedby`
- **Mark required fields** clearly and programmatically
- **Use appropriate input types** for better mobile experience
- **Provide clear error messages** with `role="alert"`

## ARIA (Accessible Rich Internet Applications)

ARIA attributes enhance HTML semantics for complex interfaces:

<InteractiveCodeBlock title="Essential ARIA Attributes" description="Using ARIA to improve accessibility:">
{`<!-- aria-label: Provides accessible name -->
<button aria-label="Close dialog">√ó</button>

    Menu
</button>

    <li><a href="/about">About</a></li>
</ul>

<!-- aria-hidden: Hides decorative content from screen readers -->
<span aria-hidden="true">üëç</span> Great job!

<!-- aria-live: Announces dynamic content changes -->

<div aria-live="assertive" id="error-announcement"></div>

    Please fix the errors below

## Keyboard Navigation

Many users navigate with keyboards only, so all interactive elements must be keyboard accessible:

<InteractiveCodeBlock title="Keyboard Accessibility" description="Ensuring proper keyboard navigation:">
{`<!-- ‚úÖ Keyboard accessible elements -->

<!-- Skip link for keyboard users -->
<a href="#main-content" class="skip-link">Skip to main content</a>

<!-- Custom dropdown with keyboard support -->
<div class="custom-select" 
     tabindex="0" 
     role="combobox" 
     aria-expanded="false"
     aria-haspopup="listbox"
     onkeydown="handleKeydown(event)">
    Select an option
</div>

<!-- Button that looks like a link -->
<button class="link-button" 
        onclick="navigate()"
        onkeydown="handleEnterSpace(event)">
    Go to next page
</button>

<!-- Focus management in modals -->
<div role="dialog" aria-modal="true">
    <button class="close-button" 
            onclick="closeModal()"
            aria-label="Close dialog">√ó</button>
    <h2>Modal Title</h2>
    <p>Modal content...</p>
    <button>Action</button>
    <button onclick="closeModal()">Cancel</button>
</div>

<!-- Accessible carousel -->
<div class="carousel" role="region" aria-label="Featured articles">
    <button aria-label="Previous article" onclick="previousSlide()">‚Äπ</button>
    <div role="img" aria-label="Article 1 of 5: Web Accessibility Basics">
        <!-- slide content -->
    </div>
    <button aria-label="Next article" onclick="nextSlide()">‚Ä∫</button>
    
    <div role="tablist" aria-label="Choose article">
        <button role="tab" aria-selected="true">1</button>
        <button role="tab" aria-selected="false">2</button>
        <button role="tab" aria-selected="false">3</button>
    </div>
</div>

<!-- ‚ùå Keyboard inaccessible -->

<!-- Div that acts like a button but can't be reached by keyboard -->

<!-- Links that don't indicate they open new windows -->
<a href="external-site.com">Visit our partner</a>

<!-- Custom elements without proper focus management -->
<div class="dropdown">
    <div onclick="toggleMenu()">Menu</div>
    <div class="menu-items">
        <div onclick="selectItem()">Item 1</div>
        <div onclick="selectItem()">Item 2</div>
    </div>
</div>`}
</InteractiveCodeBlock>

### Keyboard Navigation Requirements
- **All interactive elements** must be focusable
- **Focus indicators** must be visible and clear
- **Tab order** should be logical and intuitive
- **Enter/Space** should activate buttons and links
- **Arrow keys** should navigate within widgets
- **Escape** should close modals and dropdowns

        ‚úó Please enter a valid email address

<!-- ‚ùå Poor contrast and color-only communication -->

    background: #cccccc;
    color: #dddddd; /* 1.2:1 - fails WCAG guidelines */
}

    color: red; /* Information conveyed by color alone */
}
</style>`}
</InteractiveCodeBlock>

### Color Guidelines
- **Normal text**: 4.5:1 contrast ratio minimum
- **Large text** (18pt+ or 14pt+ bold): 3:1 contrast ratio minimum
- **Don't use color alone** to convey information
- **Test with color blindness** simulators
- **Provide multiple indicators** (color, text, icons, patterns)

## Common Accessibility Mistakes

### 1. Missing Alternative Text
```html
<!-- ‚ùå Bad -->
<img src="chart.png">

<!-- ‚úÖ Good -->
<img src="chart.png" alt="Sales increased 25% from Q1 to Q2">
```

### 2. Poor Color Contrast
```css
/* ‚ùå Bad - 2.1:1 contrast */
.text { color: #999; background: #fff; }

/* ‚úÖ Good - 7.7:1 contrast */
.text { color: #333; background: #fff; }
```

### 3. Inaccessible Forms
```html
<!-- ‚ùå Bad -->
<input type="text" placeholder="Name">

<!-- ‚úÖ Good -->
<label for="name">Name</label>
<input type="text" id="name" name="name">
```

### 4. Non-Semantic Interactive Elements
```html
<!-- ‚ùå Bad -->
<div onclick="submit()">Submit</div>

<!-- ‚úÖ Good -->
<button type="submit">Submit</button>
```

### 5. Missing Focus Management
```html
<!-- ‚ùå Bad - No visible focus indicator -->
<button style="outline: none;">Click me</button>

<!-- ‚úÖ Good -->
<button class="has-focus-indicator">Click me</button>
```
